## Objective

Close the implementation and coverage gaps between mlx_atomistic and production-ready MD simulation engines (OpenMM, LAMMPS), adding all missing capabilities needed for real biomolecular MD simulations on Apple Silicon.

## Broader Intent

Make mlx_atomistic a credible alternative to OpenMM for atomistic simulation on Apple Silicon — not just a research prototype, but a production-capable engine where a user can set up and run a real protein+ligand simulation with proper force fields, thermodynamics, and checkpointing without hitting a "not implemented" wall.

## Shape Context: Closure Target

Reference systems: OpenMM 8.x (primary), LAMMPS (secondary for breadth).

Gap landscape (ordered by priority for biomolecular MD):

1. **Minimization**: Only steepest descent — missing L-BFGS, conjugate gradient, FIRE
2. **Constraints**: Only iterative distance constraints — missing SETTLE (analytical water), CCMA (parallel arbitrary constraints), HMR
3. **Periodic boundary conditions**: Only orthorhombic — missing triclinic
4. **Barostats**: Only isotropic MC — missing anisotropic MC, membrane (surface tension) MC, flexible triclinic MC
5. **Thermostats**: Only Langevin BAOAB — missing Nosé-Hoover, Andersen, Berendsen, velocity rescaling (Bussi CSVR)
6. **Virtual sites**: No framework — missing mass-weighted averages, out-of-plane, local-coordinate sites, force redistribution, TIP4P/5P lone pairs
7. **Hydrogen mass repartitioning**: Not implemented
8. **CMAP**: Implemented but needs validation reference against OpenMM CMAP
9. **Implicit solvent**: No GBSA/OBC or custom GB — critical gap for drug design
10. **Alchemical free energy**: No lambda-scaled potentials, soft-core, ATM, or alchemical factory
11. **Replica exchange**: No multi-copy or exchange infrastructure
12. **Custom force expressions**: No algebraic custom force framework
13. **Multiple time stepping**: No r-RESPA or force group separation
14. **Force field parsers**: No AMBER prmtop, CHARMM PSF/params, GROMACS top/gro readers (prep/ has partial GPCRMD import)
15. **PME higher-order**: Only CIC (order 2) — missing 4th/5th order B-spline
16. **Ryckaert-Bellemans dihedral**: Not implemented (CHARMM compatibility)
17. **Polarizable force fields**: No Drude or AMOEBA support
18. **Enhanced sampling**: No metadynamics, umbrella sampling, or ABF beyond steered MD
19. **RMSD restraints**: Not implemented
20. **Triclinic PBC**: Not implemented (needed for membrane, non-orthogonal cells)

What "closed" looks like: A researcher can set up, run, and validate a protein+ligand simulation with CHARMM36 or AMBER ff14SB force fields, explicit or implicit solvent, proper thermodynamics (NVT, NPT with appropriate barostat), constraints including SETTLE/HMR, and compare results against OpenMM with confidence that the physics matches.

## Constraints

- Python 3.13 only, MLX/Metal as GPU backend
- No CUDA or non-Apple-Silicon GPU support
- No heavyweight chemistry packages without concrete need (CLAUDE.md rule)
- Must maintain OpenMM parity validation for every new capability
- vendors/ stays reference-only — do not import from vendor trees
- Ruff lint must pass (E, F, I, UP, B, SIM, line-length 100)

## Scope Coverage

- Included: All 20 gap items listed above, phased for implementation
- Deferred: Polarizable force fields (Drude/AMOEBA) — very high complexity, growing but not yet mainstream demand; can be a later phase
- Anti-goals: CUDA/ROCm support; replacing OpenMM or LAMMPS for x86 HPC; reactive force fields (ReaxFF); machine-learned potentials; materials science potentials (EAM, MEAM, Tersoff, SW)
- Needs decision: Whether GBSA/OBC is Phase 2 or Phase 3 priority; whether custom force expressions come via algebraic parsing or programmatic API

## Key Assumptions and Risks

- Virtual sites require force redistribution architecture touching nonbonded and constraint code — highest implementation risk
- GBSA/OBC requires accurate surface area calculation on GPU — nontrivial MLX kernel work
- Force field parsers are tedious but architecturally simple — low risk, high effort
- Some gaps (triclinic PBC, anisotropic barostat) have cascading effects on cell handling throughout the codebase
- Reference validation against OpenMM requires OpenMM to be available in dev environment (already a dev dependency)

## Deferred Scope

- Polarizable force fields (Drude, AMOEBA) — roadmap Phase 6+ or later
- Machine-learned potentials (GAP, ACE, NequIP, etc.)
- Reactive force fields (ReaxFF)
- Materials science potentials (EAM, MEAM, Tersoff, Stillinger-Weber)
